## Findings

| ID | Category | Severity | Finding | Evidence (file:line) | Recommendation |
|---|---|---|---|---|---|
| F01 | Security | Critical | `.env` is committed to git and is **not** listed in `.gitignore`. Today it contains only the Supabase URL and the `anon` (publishable) key, but the policy of tracking `.env` means any future addition (service-role key, Resend key, Gemini key) will leak by default. | `.env:1-3` — `VITE_SUPABASE_PUBLISHABLE_KEY="eyJhbGciOiJIUzI1NiIs..."`; `git ls-files` returns `.env`; `.gitignore:1-25` contains no `.env` entry | Add `.env`, `.env.*`, `!.env.example` to `.gitignore`. `git rm --cached .env`. Provide a committed `.env.example` with placeholder values. Rotate any key that has ever been in the file. |
| F02 | Security | Critical | Postgres RLS is "enabled" on every table, but every policy is `USING (true) WITH CHECK (true)` — i.e. fully open. Combined with the `anon` key being effectively public (shipped to the browser), **any internet caller can read/insert/update/delete every row in every table.** The migration comment explicitly admits this ("Allow public access (no auth required for this app)"). | `supabase/migrations/20260103043037_*.sql:87-101` — `CREATE POLICY "Allow all access to sessions" ON public.free_agent_sessions FOR ALL USING (true) WITH CHECK (true);` (repeated for `blackboard`, `tool_calls`, `artifacts`, `session_files`, `messages`) | Introduce Supabase auth (or a server-side session key), add an `owner_id uuid` to every table, and replace each policy with `USING (auth.uid() = owner_id)`. Until then, do not put real user content in these tables. |
| F03 | Security | Critical | All 22 Edge Functions have `verify_jwt = false`. Combined with `Access-Control-Allow-Origin: *` (F08), every function (including `send-email`, `external-db`, `api-call`, `run-agent`) is an unauthenticated public endpoint. | `supabase/config.toml:3-67` — every function block: `[functions.<name>]\nverify_jwt = false` (22 blocks) | Flip `verify_jwt = true` for any function that touches paid APIs, the database, outbound mail, or arbitrary URLs. For functions that genuinely need to be public, gate them with an HMAC-signed token, IP allowlist, or per-route rate limit. |
| F04 | Security | Critical | **Server-Side Request Forgery in `api-call`.** Accepts an arbitrary URL from the request body and fetches it server-side, returning the full response. No protocol allowlist (`file://`, `http://localhost`, `http://169.254.169.254/...` not blocked), no host allowlist, no DNS-rebinding defense. Anonymous (F03), so a public attacker can pivot to cloud-metadata endpoints, internal Supabase services, or any HTTP service the function's egress can reach. | `supabase/functions/api-call/index.ts:59-86` — `const { url, method = "GET", headers = {}, body } = await req.json(); ... const response = await fetch(url, fetchOptions);` | Reject any URL whose scheme is not `https:`; resolve the host, reject if it resolves to RFC1918, loopback, link-local (`169.254.0.0/16`), or `fc00::/7`; consider an explicit allowlist for known APIs. Add `verify_jwt = true`. |
| F05 | Security | Critical | **SSRF in `web-scrape`** — same shape as F04, plus the function deliberately rotates User-Agent strings and fakes `Referer: https://www.google.com/` to evade bot detection on target sites (potential ToS/legal exposure on top of the SSRF). | `supabase/functions/web-scrape/index.ts:99-130` (UA rotation, faked referer); line 301 `const { url, returnHtml, maxCharacters } = await req.json();` then 350 `response = await smartFetch(url, 0, ...)` | Same URL validation as F04. Remove UA-spoofing / fake referer — set an honest `User-Agent: AgentBuilderConsole/1.0 (+contact)`. Respect `robots.txt`. |
| F06 | Security | Critical | **`external-db` is an open arbitrary-SQL proxy.** Accepts a Postgres `connectionString` and `query` from the request body and executes the SQL with no host allowlist, no read-only enforcement (`isWrite` is descriptive, not enforced — writes will run either way), no statement allowlist, no rate limit, and no auth. | `supabase/functions/external-db/index.ts:276-300` — `const { connectionString, action = "query", query, isWrite = false, params = [] } = requestBody; ... const client = new Client(secureConnectionString); await client.connect();`; line 375 `const result = await client.queryArray(stmt, params);` | Remove the function or restrict it to a connection string the function builds itself from server-side secrets. If user-supplied connection strings must be supported, gate with auth + host allowlist + `SET TRANSACTION READ ONLY` for reads + explicit `BEGIN; ...; ROLLBACK;` for ad-hoc queries. |
| F07 | Security | Critical | **Open email relay.** `send-email` accepts any recipient, subject and body, with no auth (F03), no rate limit, no recipient allowlist, no per-IP throttling. Sender is hard-coded to `Agent Builder Console <info@agentbuilderconsole.com>`. Anyone on the internet can send mail from that address while it lasts in Resend. | `supabase/functions/send-email/index.ts:23-52` — `const resend = new Resend(Deno.env.get("RESEND_API_KEY")); const { to, subject, body, useHtml }: SendEmailRequest = await req.json(); ... await resend.emails.send({ from: "Agent Builder Console <info@...>", to: [to], subject, ...(useHtml ? { html: body } : { text: body }) });` | Require auth, restrict `to` to verified addresses or a small allowlist, add IP-based rate limiting (e.g. 5 req / hour / IP), strip HTML by default, and alert on abuse spikes. |
| F08 | Security | High | Wildcard CORS (`Access-Control-Allow-Origin: *`) on every edge function. Allows any website to invoke these endpoints from a visitor's browser, which — combined with F03 / F04 / F06 / F07 — amplifies the blast radius (drive-by SSRF, drive-by SQL, drive-by spam). | `supabase/functions/*/index.ts` — pattern `'Access-Control-Allow-Origin': '*'` matches in all 22 functions. E.g. `supabase/functions/run-agent/index.ts:4` — `'Access-Control-Allow-Origin': '*',` | Replace `*` with the production origin (e.g. `https://agentbuilderconsole.com`) and the local dev origin behind a `mode === "development"` guard. For functions that legitimately serve third parties, keep `*` but require auth (F03). |
| F09 | Security | High | Gemini safety filters disabled globally (`BLOCK_NONE` for harassment, hate, sexually explicit, and dangerous content). Combined with F03 this is an anonymous, content-unmoderated LLM endpoint. | `supabase/functions/run-agent/index.ts:163-180` — `safetySettings: [ { category: "HARM_CATEGORY_HARASSMENT", threshold: "BLOCK_NONE" }, { category: "HARM_CATEGORY_HATE_SPEECH", threshold: "BLOCK_NONE" }, { category: "HARM_CATEGORY_SEXUALLY_EXPLICIT", threshold: "BLOCK_NONE" }, { category: "HARM_CATEGORY_DANGEROUS_CONTENT", threshold: "BLOCK_NONE" } ]` | Default to `BLOCK_MEDIUM_AND_ABOVE`. If looser thresholds are needed for a specific feature, scope the override to that code path and gate it behind auth + audit logging. |
| F10 | Security | High | Supabase session is stored in `localStorage` with `persistSession: true`. Any XSS (including via a future regression of F16 or a vulnerable dependency in the large UI surface) exfiltrates the JWT, which lets the attacker act as the user against the (currently wide-open, see F02) database. | `src/integrations/supabase/client.ts:11-16` — `export const supabase = createClient<Database>(SUPABASE_URL, SUPABASE_PUBLISHABLE_KEY, { auth: { storage: localStorage, persistSession: true, autoRefreshToken: true } });` | Once auth is in place, prefer `@supabase/ssr` with httpOnly cookies. At minimum, add a strict `Content-Security-Policy` header (currently absent from `index.html`) to limit XSS reach. |
| F11 | Security | High | Error responses leak server stack traces to the client. Useful for fingerprinting Deno version, file layout, and dependency chain. | `supabase/functions/run-agent/index.ts:309-318` — `return new Response(JSON.stringify({ error: errorMessage, stack: errorStack, details: 'Full error details are available in the edge function logs' }), { status: 500, ... });` ; same pattern in `supabase/functions/run-agent-anthropic/index.ts:236-244` — `details: error instanceof Error ? error.stack : undefined` | Return a generic `{ error: "Internal error", requestId }` to the client; log the stack server-side keyed on `requestId`. |
| F12 | Security | High | Third-party API keys are passed from the browser through edge functions as plain JSON config. The architecture is "user pastes their Google/Anthropic/xAI key into the UI, app stores it in `sessionStorage`, app POSTs it to the edge function in the request body." Any XSS, any page-injected extension, any user with browser devtools, and the Supabase log retention pipeline (F14) can read those keys. | `supabase/functions/run-agent/index.ts:53-57` — `body: JSON.stringify({ query: userPrompt, apiKey: config?.apiKey, searchEngineId: config?.searchEngineId })`; line 64-68 (`apiKey: config.apiKey`); `src/hooks/useSecretsManager.ts:38-44` — `sessionStorage.setItem(STORAGE_KEY, JSON.stringify(config))` | Keep third-party keys server-side as `Deno.env.get(...)` (the function already does this for `GEMINI_API_KEY` on line 31 — extend the pattern). If users truly need to bring their own keys, encrypt at rest with a key derived from the user's password and never log the value (F14). |
| F13 | Security | High | Secrets (API keys, OAuth tokens, custom headers) are stored unencrypted in `sessionStorage`. They survive every reload of the tab and are exfiltratable by any JavaScript that runs in the page. | `src/hooks/useSecretsManager.ts:38-44` — `function saveToStorage(config: SecretsConfig): void { try { sessionStorage.setItem(STORAGE_KEY, JSON.stringify(config)); } catch (e) { console.warn('Failed to save secrets to sessionStorage:', e); } }` ; line 56-69 — stores `value: string` as-is | If keys must live client-side, encrypt with `SubtleCrypto` using a passphrase-derived key the user enters each session and never persist the passphrase. | F14 | Privacy | Medium | User prompts, recipient emails, SQL statements, and tool outputs are written to function logs in clear text. These logs go to Supabase log storage with unknown retention. PII (the user's prompt may contain anything), API keys embedded in URLs (e.g. `?apiKey=...`), and recipient email addresses are all captured. | `supabase/functions/run-agent/index.ts:36` — `console.log("Running agent with system prompt:", systemPrompt.substring(0, 50));` and line 60-72 (logs each tool output) ; `supabase/functions/send-email/index.ts:26` — `console.log(\`Sending email to: ${to}, subject: ${subject}\`);` ; `supabase/functions/external-db/index.ts:372` — `console.log(\`[External DB] Statement ${i + 1}/${statements.length}: ${stmt.slice(0, 100)}...\`);` ; `supabase/functions/api-call/index.ts:65` — `console.log(\`Making ${method} request to:\`, url);` (URL may contain query-string credentials) | Redact: never log full prompts, recipient addresses, full URLs, or SQL bodies. | F15 | Privacy | Low | The `from` address for outbound mail is hard-coded to a single sender domain — fine functionally, but the `useHtml` flag lets a caller send raw HTML bodies under that brand. Combined with F07 this is a phishing kit. | `supabase/functions/send-email/index.ts:47-52` — `from: "Agent Builder Console <info@agentbuilderconsole.com>", ... ...(useHtml ? { html: body } : { text: body })` | Strip / sanitize HTML (e.g. `DOMPurify`) before sending, or remove the `useHtml` flag entirely and only ever send `text`. |
| F16 | Security | Medium | `dangerouslySetInnerHTML` is used to inject a `<style>` block whose CSS values come from a `config` object. Today the values come from internal `THEMES` (safe). But the function accepts `id`, `key` and `color` from a `ChartConfig` prop — if a future caller sources that prop from saved-workflow JSON (which is already loaded from `localStorage` and import-from-file, see `src/components/sidebar/Sidebar.tsx`), CSS-injection / `</style><script>` breakout becomes a real path. | `src/components/ui/chart.tsx:69-86` — `<style dangerouslySetInnerHTML={{ __html: Object.entries(THEMES).map(([theme, prefix]) => \`...\${prefix} [data-chart=${id}] { ... --color-${key}: ${color}; ... }\`).join("\n") }} />` | Validate `id` against `^[a-zA-Z0-9_-]+$`, validate each `color` against a CSS-color regex, and reject anything containing `<`, `>`, `{`, `}`. Better: emit a `<style>` element whose `textContent` is set imperatively (React won't serialize HTML tags through `textContent`). |
| F17 | Security | Medium | Edge functions don't validate request bodies at runtime — they only TS-cast `await req.json()` to an interface. Zod is in `dependencies` (`zod ^3.25.76`) but isn't used in any function. Missing or wrong-type fields are caught only when something downstream throws, often after side effects (a DB connection has been opened, an outbound fetch has fired). | `supabase/functions/send-email/index.ts:24` — `const { to, subject, body, useHtml }: SendEmailRequest = await req.json();` ; `supabase/functions/api-call/index.ts:59` — `const { url, method = "GET", headers = {}, body } = await req.json();` ; `supabase/functions/external-db/index.ts:276-284` — `const requestBody: ExternalDbRequest = await req.json(); const { connectionString, action = "query", query, isWrite = false, params = [], schemas: schemaFilter } = requestBody;` | Add a Zod schema at the top of each handler and `schema.parse(await req.json())` before any branching. | F18 | Security | Medium | No `Content-Security-Policy`, `X-Frame-Options`, `Referrer-Policy`, or `Permissions-Policy` headers are set on the SPA. The HTML shell only has OG/Twitter meta tags. | `index.html:1-29` — no `<meta http-equiv="Content-Security-Policy" ...>` and no equivalent header config in `vite.config.ts:1-18` | Add a CSP via Vite middleware or your hosting provider. Minimum: `default-src 'self'; script-src 'self'; object-src 'none'; frame-ancestors 'none'; base-uri 'self'`. |
| F19 | Reliability | Medium | The `run-agent` Gemini call sets `safetySettings: BLOCK_NONE` on a model that may surface in user-visible chat. Beyond F09's security angle, this also means transient model-level changes (Google tightening defaults) won't be caught by your fallback — the function will keep emitting whatever the model produces. | `supabase/functions/run-agent/index.ts:163-180` (same evidence as F09) | (See F09 — same fix.) | F20 | Reliability | Medium | `external-db` opens a fresh Postgres connection per request (`new Client(secureConnectionString); await client.connect();`) and closes it in `finally`. Under load this exhausts target-side connection limits and adds ~100ms+ per call. There's also no statement timeout — a slow query stalls the Edge Function until the platform kills it. | `supabase/functions/external-db/index.ts:299-300, 459-461` — `const client = new Client(secureConnectionString); await client.connect(); try { ... } finally { await client.end(); }` | After connect, run `SET statement_timeout = '15s'`. Consider connection pooling via Supavisor on the target db, and document that callers should use pgbouncer-style URLs. |
| F21 | Reliability | Medium | `web-scrape` retries with backoff on every error class, including ones unlikely to recover (HTTP 4xx, schema errors). Worst case 3 retries × 30s timeout × 8s backoff ≈ over a minute per scrape, multiplied per concurrent caller. | `supabase/functions/web-scrape/index.ts:132-241` — `const maxRetries = 3; ... if (!response.ok && retryCount < maxRetries) { ... return smartFetch(url, retryCount + 1, options); }` | Don't retry on `4xx`. Cap total wall-clock time per request to a single budget (e.g. 20s) using a shared `AbortSignal.timeout`. |
| F22 | Maintainability | Medium | Both `package-lock.json` (387 KB) and `bun.lockb` (197 KB) are checked in, but the `scripts` block only invokes `vite`/`eslint` — there's no documented install command. New contributors don't know whether `npm i`, `bun i`, or `pnpm i` is canonical, and the two lockfiles will drift. | `package.json:6-12` (scripts) ; `bun.lockb` (binary, 197327 bytes) ; `package-lock.json` (387329 bytes) | Pick one (Bun, per `bun.lockb` and the `lovable-tagger` ecosystem), delete the other, and add a `"engines"` field plus a one-liner in `README.md`. |
| F23 | Maintainability | Low | `error: any` catch blocks across edge functions defeat TypeScript's narrowing and make stack-leak fixes (F11) harder to grep for. | `supabase/functions/send-email/index.ts:80` — `} catch (error: any) {` ; `supabase/functions/run-agent-anthropic/index.ts:234-244` (uses `error instanceof Error` but still surfaces `.stack` to client) | Use `catch (error: unknown)` + `error instanceof Error` narrowing, and forbid `any` via `eslint`. |
| F24 | Maintainability | Low | Duplicate weather functions: `supabase/functions/weather/index.ts` and `supabase/functions/tool_weather/index.ts` both exist and are both registered in `config.toml`. Two surfaces, two sets of CORS/JWT settings to keep in sync. | `supabase/config.toml:12-13` (`[functions.weather]\nverify_jwt = false`) and `:63-64` (`[functions.tool_weather]\nverify_jwt = false`) | Delete the unused one. | F25 | Performance | Low | Every edge function pulls `https://deno.land/std@0.168.0/http/server.ts` — a version from mid-2023. Each cold start re-resolves and re-bundles that URL. ## Category coverage summary

- **Security:** F01–F13, F16–F18 (16 findings) — `Critical`/`High` density driven by F02 (open RLS), F03 (no JWT verification), and F04–F07 (SSRF, open SQL proxy, open email relay).
- **Privacy:** F14–F15 (2 findings) — log leakage and unsanitized HTML mail bodies.
- **Performance:** F25 (1 finding) — outdated Deno `std` pinning.
- **Reliability:** F19–F21 (3 findings) — disabled safety filters, no statement timeout, over-eager retries.
- **Maintainability:** F22–F24 (3 findings) — dual lockfiles, `error: any`, duplicate weather function.

No category was empty.

## Suggested triage order

1. **F01, F03** — easy wins that close large attack surfaces immediately (gitignore + revoke + flip `verify_jwt`).
2. **F02** — requires schema change (`owner_id`) and an auth design; until it's done, treat the DB as a public spreadsheet.
3. **F04, F05, F06, F07** — each is a single-function rewrite (URL/host validation, read-only enforcement, recipient allowlist).
4. **F11, F14** — small diffs across many functions; do them as one sweep.
5. **F09, F19, F16, F17, F18** — defensive hardening once the bleeding has stopped.
6. **F22–F25** — maintenance.
